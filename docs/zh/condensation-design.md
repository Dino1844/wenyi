# 书本浓缩流水线设计文档

## 项目定位

将 Wenyi（文学翻译工具）改造为通用书籍浓缩工具。输入一本 EPUB/PDF/TXT，输出一本可独立阅读的浓缩版 EPUB。浓缩不是概括——不是告诉读者"这本书讲了什么"，而是让读者以最短路径经历这本书的核心论证/叙事/知识推进。

核心原则：浓缩版应当保留"跟着走一遍"的阅读体验，而非"被告知结论"的报告感。

---

## 流水线总览

```
读取输入（复用 ingest/）
→ 切分章节与段落
→ 书籍类型识别（论证/叙事/知识/混合）
→ 全书结构分析（论证链/情节弧/知识图谱）
→ 建立概念图（关键概念 + 书中定义）
→ 逐章角色定位（这章在全书结构中承担什么功能）
→ 逐章浓缩（按类型选策略，保留推进感）
→ 流畅度润色（去报告感，确保论证/叙事的惯性）
→ 逻辑审校（检查关键步骤是否丢失、概念是否自足）
→ 组装浓缩版 EPUB
```

---

## 阶段一：书籍类型识别

### 职责
判断输入书籍的主导类型，决定后续浓缩策略。

### 输入
全书前 N 章样本（复用现有 `_sample_text` 的多点采样逻辑）。

### 输出
```json
{
  "book_type": "argumentative | narrative | knowledge | mixed",
  "sub_type": "哲学论证 | 科普 | 回忆录 | 小说 | 教材 | 自助 | ...",
  "condensation_strategy": "preserve_argument_chain | preserve_plot_arc | preserve_concept_map | hybrid",
  "reasoning": "判断依据（2-3句）"
}
```

### 类型与策略映射

| 类型 | 浓缩策略 | 保留什么 | 去掉什么 |
|------|----------|----------|----------|
| 论证类（哲学、学术论辩） | 保留推理链 | 核心论证、关键类比、思想实验、"被击中"的瞬间 | 学术辩护、文献综述、重复论证、对同行的逐条回应 |
| 叙事类（小说、回忆录） | 保留情节弧 | 关键场景、人物转折、对话精华、情感高点 | 环境描写铺陈、过渡段、次要支线 |
| 知识类（科普、教材、自助） | 保留概念图 | 核心概念及其定义、关键例子（每概念一个）、概念间关系 | 重复举例、过渡性废话、"如前所述"式回顾 |
| 混合类 | 逐章判断 | 按每章的主导模式选策略 | 同上 |

### 提示词方向
```
你是一位阅读策略分析师。阅读以下书籍样本，判断这本书的类型和最佳浓缩策略。
不要只看书名或主题——要看它的写作方式：它是在论证一个观点、讲述一个故事、
还是传授一套知识？它的核心"推进力"是什么——逻辑推理、情节张力、还是概念层层展开？
```

## 阶段二：全书结构分析

### 职责
替代原来的 Analyzer（风格分析）。不是分析"怎么写"，而是分析"在做什么"——全书的论证/叙事/知识结构是什么。

### 输入
多点采样文本 + 目录结构（如果 EPUB 有 TOC）。

### 输出
```json
{
  "central_question": "这本书试图回答的核心问题",
  "structure_type": "linear_argument | layered_concepts | chronological | problem_solution | ...",
  "arc": "全书推进的弧线描述（3-5句）",
  "key_moves": [
    {"chapter_range": "1-12", "function": "建立通用思维工具"},
    {"chapter_range": "13-33", "function": "处理'意义'问题"},
    ...
  ],
  "concepts": [
    {"term": "感受质", "definition_in_book": "体验中独立于一切功能关系的内禀性质", "first_appears": "ch56"},
    {"term": "意向立场", "definition_in_book": "把实体当作有信念和欲望的行动者来预测的策略", "first_appears": "ch18"},
    ...
  ]
}
```

### 提示词方向
```
你是一位结构分析师。你的任务不是总结这本书"讲了什么"，而是画出它的骨架：
- 它在回答什么问题？
- 它的论证/叙事/知识是按什么顺序推进的？
- 哪些部分是"承重墙"（去掉它全书塌掉），哪些是"装饰"（去掉不影响结构）？
- 关键概念在哪里被定义、在哪里被使用？
```

## 阶段三：概念图（替代术语表）

### 职责
追踪全书关键概念，确保浓缩版中每个概念在首次出现时有足够定义，不依赖读者读过原文。

### 与翻译项目术语表的区别
- 术语表：source → target 的映射（确保译名一致）
- 概念图：term → definition_in_context 的映射（确保概念自足）

### 数据结构
```python
@dataclass
class ConceptEntry:
    term: str                    # 概念名
    definition: str              # 本书中的定义/用法（浓缩版应包含的）
    first_chapter: int           # 首次出现的章节
    role: str                    # 在全书论证中的角色
    dependencies: list[str]      # 理解它需要先理解哪些概念
```

### 运行时行为
- 结构分析阶段建立初始概念图
- 每章浓缩后，从浓缩文本中抽取新出现的概念，更新概念图
- 后续章节浓缩时，注入"本章涉及的概念及其定义"，确保浓缩文本中概念有交代

## 阶段四：逐章角色定位（替代章节梗概）

### 职责
为每一章确定它在全书结构中的"功能角色"——不是"这章讲了什么"，而是"这章在全书推进中干了什么活"。

### 输入
章节原文 + 全书结构分析结果。

### 输出
```json
{
  "chapter_index": 57,
  "role": "用类比论证拆解感受质的实在性",
  "argument_function": "正面论证（用'美元活力'类比证明感受质是虚构）",
  "key_passages": ["第3-5段（类比展开）", "末段（反问落点）"],
  "dispensable": ["第1段（对内格尔论文的学术回应）", "第6段（对经济学家的冗余解释）"],
  "concepts_used": ["感受质", "内禀性质", "功能关系"],
  "concepts_introduced": ["美元的活力（类比概念）"]
}
```

### 提示词方向
```
你是一位编辑，正在为一本浓缩版做章节规划。读完这一章，告诉我：
1. 它在全书论证/叙事中承担什么功能？（如果删掉这章，全书会缺什么？）
2. 哪些段落是"泵"——真正推动读者认知前进的核心段落？
3. 哪些段落是"说明书"——学术辩护、重复、过渡、文献回顾？
4. 如果只能保留原文的30%，你保留哪些段落？为什么？
```

## 阶段五：逐章浓缩（替代翻译）

### 职责
核心阶段。接收章节原文 + 角色定位 + 概念图 + 全书结构，输出浓缩文本。

### 输入
- 章节原文
- 该章的角色定位（阶段四输出）
- 全书结构分析（阶段二输出）
- 本章涉及的概念及定义（概念图子集）
- 前文浓缩文本尾部（滚动上下文，保持衔接）
- 浓缩策略（由书籍类型决定）

### 输出
浓缩后的文本（目标长度：原文的 15%-30%，具体由 config 控制）。

### 浓缩策略细则

**论证类：**
```
你的任务不是"总结这一章的观点"，而是让读者"跟着论证走一遍"。
规则：
1. 保留推理的推进感——每一步必须从上一步逼出来，不能跳跃。
2. 保留核心思想实验/类比的完整展开（这是"泵"本身）。
3. 去掉：对同行论文的引用和回应、"正如我在第X章所说"式回顾、
   同一论点的第二第三个补充例子（保留最有力的一个）。
4. 概念首次出现时，用一句话给出本书中的定义，不假设读者已知。
5. 不用"作者认为""作者指出"这类转述语——让论证自己说话。
6. 结尾保留原文的"落点"（通常是一个反问或一个不舒服的结论）。
```

**叙事类：**
```
你的任务不是"缩写故事"，而是保留读者经历这个故事的情感弧线。
规则：
1. 保留关键场景的完整性（对话、动作、转折）。
2. 用一两句过渡连接场景之间的空白，不铺陈。
3. 保留人物的声音——关键对话原样保留，不改为间接引语。
4. 去掉：环境描写的细节铺陈、次要支线的展开、重复的情感渲染。
5. 保留"意外"——叙事中让读者吃惊的时刻必须完整保留。
```

**知识类：**
```
你的任务不是"提取知识点"，而是让读者经历"概念层层展开"的过程。
规则：
1. 每个核心概念：给出定义 + 一个最有力的例子/类比。
2. 保留概念之间的连接——"为什么理解A之后才能理解B"。
3. 去掉：同一概念的多个重复举例、"让我们回顾一下"式段落、
   过渡性废话（"接下来我们将讨论……"）。
4. 保留"啊哈时刻"——概念突然变得清晰的那个例子或类比。
```

### 提示词模板（论证类示例）
```
【全书结构】
$book_structure

【本章角色】
$chapter_role

【概念图（本章涉及）】
$concepts

【前文浓缩（最近）】
$context

【章节原文】
$source

【浓缩策略】
$condensation_rules

请将以上章节浓缩为一段流畅的论证文本。目标长度：原文的20%左右。
要求：保留论证的推进感，让读者"跟着走"而非"被告知"。
概念首次出现时给出定义。不用"作者认为"式转述。
输出 JSON：{"condensed": "浓缩后的文本"}
```

## 阶段六：流畅度润色（替代翻译润色）

### 职责
去除浓缩文本的"报告感"和"列表感"，确保它读起来像一条连续的思维河流。

### 输入
浓缩后的文本。

### 检查清单
- 是否有"首先……其次……最后……"的列表结构？→ 改为自然推进
- 是否有"本段讨论的是……"式的元叙述？→ 删除，直接进入论证
- 句子节奏是否过于均匀？→ 长短交替
- 段落之间的过渡是否靠"路标词"（"接下来""另外"）？→ 靠思路惯性滑过去
- 是否有"作者""本文"等第三方视角？→ 让论证/叙事自己说话

### 提示词方向
```
你是一位文体编辑。以下是一段浓缩后的论证/叙事文本。
它的问题可能是：读起来像报告、像列表、像摘要，而不像"一个人在带你走一条路"。
请在不改变内容、不增加信息的前提下，调整它的节奏和衔接，
让它读起来像一条连续的河流，而非一堆分段的要点。
```

## 阶段七：逻辑审校（替代翻译审校）

### 职责
检查浓缩版是否丢失了关键论证步骤、是否引入了逻辑跳跃、概念是否自足。

### 检查项
| 问题类型 | 说明 |
|----------|------|
| logic_gap | 论证中缺少了必要的一步，读者无法从A推到B |
| concept_undefined | 某概念被使用但未在浓缩版中定义过 |
| lost_punch | 原文的"落点"（关键反问/结论）在浓缩版中消失了 |
| false_summary | 浓缩版把论证变成了断言（"X是因为Y"但原文是逐步推出的） |
| flow_break | 前后段之间缺乏逻辑衔接，读起来像两个不相关的片段 |

### 输出
```json
{
  "issues": [
    {"index": 2, "type": "logic_gap", "detail": "从'感受质独立于功能'到'感受质不存在'之间缺少了'美元活力'类比的展开", "suggestion": "在两句之间补充类比的中间步骤"}
  ]
}
```

## 配置文件改造（config.yaml）

```yaml
# 书本浓缩配置

# ── LLM（复用现有结构）──────────────────────────────────────────────────
llm:
  provider: deepseek
  base_url: https://api.deepseek.com
  api_key_env: DEEPSEEK_API_KEY
  tiers:
    strong:
      model: deepseek-v4-pro
      options:
        thinking: true
        reasoning_effort: high
    cheap:
      model: deepseek-v4-flash
      options:
        thinking: true
    fast:
      model: deepseek-v4-flash

# ── 浓缩策略 ─────────────────────────────────────────────────────────────
condense:
  book_type: auto          # auto | argumentative | narrative | knowledge
  target_ratio: 0.2        # 目标浓缩比例（原文的20%）
  min_chapter_chars: 200   # 浓缩后每章最少字符数
  max_chapter_chars: 2000  # 浓缩后每章最多字符数
  preserve_passages: true  # 是否保留"关键段落"的原文（思想实验等）

# ── 流水线开关 ───────────────────────────────────────────────────────────
pipeline:
  structure_analysis: true   # 全书结构分析
  concept_map: true          # 概念图追踪
  chapter_role: true         # 逐章角色定位
  polish: true               # 流畅度润色
  review: true               # 逻辑审校
  rolling_context_segments: 4  # 注入的前文浓缩尾段数

# ── 输出 ─────────────────────────────────────────────────────────────────
output:
  format: epub             # epub | markdown | txt
  include_structure_map: true  # 在书首附加全书结构图
  about_page: true         # 书末附加"关于此浓缩"说明页

# ── 路径 ─────────────────────────────────────────────────────────────────
paths:
  state_dir: state
```

## 改造优先级

1. **Phase 1**：config + type_detector + structure_analyzer + chapter_role（能分析一本书的结构）
2. **Phase 2**：condenser + concepts（能浓缩单章）
3. **Phase 3**：flow_polisher + logic_reviewer（质控）
4. **Phase 4**：orchestrator 串联 + assemble 输出（端到端跑通）
5. **Phase 5**：用《直觉泵》做端到端测试，对比我们手工浓缩的 v5 版本
